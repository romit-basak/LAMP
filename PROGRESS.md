## 2026-07-13 — Volume mesh output: the viewshed's 3D *shape* as a surface (`--volume-format mesh`)

- User asked for a volume output that stores the 3D shape (surfaces/edges) instead of individual voxels. New shared module **`scripts/volume_mesh.py`** (deliberately torch-free so `volume_convert.py` keeps its light startup) + wiring: `viewshed.py --volume-format mesh` with `--mesh-style {blocky,smooth}` (per-observer and combined-union meshes, `viewshed_volume_{label}_mesh.ply` + 3D QC PNG) and `volume_convert.py --to mesh` (from a saved CSV; documented staircase caveat — its grid re-bins absolute z, the engine path is the faithful shape).
- Design: extraction runs in index space; a single `index_to_world` choke point applies the **terrain-following warp** (volume levels are heights above local ground, so vertex z = bilinear ground + level; NaN ground nearest-filled first) and the orientation flip. The flip's determinant argument (row→y axis swap cancels the descending-y step) is commented in code — the first version had it inverted and the unit test caught it as an exactly-negative volume.
- **`blocky` is the default (user's choice): exact voxel-boundary faces.** Corners dedup via doubled-integer coordinates (closed surface by construction); a fixed plan-diagonal triangulation makes the warped mesh's enclosed volume telescope to *identically* n_voxels × voxel volume, checked at ~1e-8 relative via divergence-theorem volume on centered coordinates (raw UTM magnitudes would drown float64). `smooth` = marching cubes (scikit-image was already pinned but unused — zero new deps) for presentation; its check degrades to positive-volume + a 0.5–1.05 ratio warn band since MC shaves corners.
- Verified: 16/16 unit checks (single/adjacent/diagonal-touch voxels, 970-voxel random blob, warp+mirror exactness to 9 decimals, NaN fill, MC on fragmented blob); engine id1 radius 150: 150,828 voxels → 100,624 faces, **volume 1,206,624.000 m³ exactly**, closed; smooth ratio 0.9988; converter path exact against its own re-binned grid; full-extent 3-observer run (voxel 6 / zstep 3): id1/id2/id3 + combined (136,975 union voxels) all exact; no-volume regression clean; the PLY round-trips into Blender 5.1 losslessly (`wm.ply_import`, counts intact post-validate).
- Notes: radius-free volumes span the whole DEM footprint (pre-existing behavior) — the full-extent 3-observer run took ~17 min on MPS; mesh extraction itself is seconds on top of the ray-casting. Converter outputs reuse the engine filename convention (in-place regeneration, same as `--to ply/npy`). Docs updated: README flag tables, walkthrough §3.7/§5 + pipeline tree, BLENDER.md "importing the volume mesh" note (UTM-far-from-origin caveat), CLAUDE.md data-asset/build-order rows.

## 2026-07-04 — Systemic dome roofline anchoring (supersedes per-dome nudges)

- User reported a third protruding dome (#192). Diagnostic across all 117: **10 domes rode > +0.5 m** above the rendered roof plane (worst #255 +1.44 m, #66 +1.15 m, #21 +1.00 m) — the manual 52/242 fixes were treating symptoms of a systemic anchor error. Root cause: `roof_z = ground(center) + Elevation`, while QGIS's vertex-bound extrusion renders the roof as the terrain-at-**vertices** plane; wherever the dome center sits on a local terrain high, the sphere rides above the rendered roof.
- Fix: new `roof_plane_z()` in `build_dome_layer.py` — least-squares plane through vertex ground + height, evaluated at the dome center (nodata-guarded; falls back to ground+height when <3 valid vertices). Verified: **max |ride| = 0.000 m across all 117 domes**; #192/#255/#66/#21/#258 all flush; all self-checks pass.
- The 52/242 manual **west shifts** were preserved through the regeneration (scripted CSV re-edit + `--from-inventory` rebuild); their plane-anchored z matches the earlier hand-computed values exactly (112.56 / 105.73), confirming the systemic fix generalizes the manual math. Layer classes unchanged (1/1.5/2 m) — QGIS users just toggle the dome layers.
- Terrain coloring handled in QGIS styling only (3D terrain texture = visible 2D canvas): sand pseudocolor ramp on the base DEM + map theme + texture size 2048; buildings/domes stay user-chosen `#fbbc4a`. No code.

## 2026-07-04 — Dome spheres clamped to their own walls (viz bug fix)

- User spotted spheres in the QGIS 3D view poking out of side walls. Diagnosis: **52 of 117** domes had `radius_m` larger than the wall clearance at their own center — `build_dome_layer.py` clamped the ortho-measured radius by the **polylabel inradius** (clearance at the footprint's *widest* point), but the detected dome center usually sits elsewhere, so the bound was too generous. ## 2026-07-04 — `--domes` flag (ray-cast surface) + root README.md

- `scripts/viewshed.py` can now bake dome caps into the ray-casting surface: new `--domes` flag (off by default) + `apply_dome_overlay()` raises the in-memory DEM array with hemispherical caps read from `dome_inventory.csv`, before `HeightfieldScene` is constructed.
